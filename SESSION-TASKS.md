## 会话 1：图片/截图识别 + 判断对错

### 目标
学生在做题对话中可以发送手写过程图片或截图，系统识别后 AI 判断对错并继续引导。

### 技术方案

#### 1.1 对话框添加图片上传入口
- 文件：`src/app/solve/page.tsx`
- 在输入框区域（约第815行）添加一个图片上传按钮（📎 或 ImagePlus 图标），位于发送按钮左侧
- 支持两种方式：点击上传 + 粘贴（Ctrl+V）
- 上传后显示缩略图预览，可以删除后重新上传
- 图片先压缩（复用 `@/lib/image-utils` 的 `compressImage`），限制最大 2MB
- 发送时将图片转为 base64，通过 API 传给后端

#### 1.2 OCR 识别路由
- 文件：新建 `src/app/api/student-image/route.ts`
- **双引擎策略**：
  1. 先尝试百度 OCR（API Key 已配置在 .env.local：`BAIDU_OCR_API_KEY` + `BAIDU_OCR_SECRET_KEY`）
  2. 如果百度 OCR 识别结果为空或质量差（文字少于5个字符），fallback 到 qwen 视觉模型（`qwen-vl-max`）
- 百度 OCR 用的是通用文字识别 API：https://aip.baidubce.com/rest/2.0/ocr/v1/general_basic
  - 先用 API Key + Secret Key 获取 access_token
  - 然后调用识别接口
- Qwen 视觉模型用已有的 qwen-vl 调用方式（参考 `src/app/api/vision-qwen/route.ts`）
- 识别 prompt：`请识别这张图片中的数学表达式和文字，保持数学符号和公式的准确性，用纯文本输出。如果是手写内容，请将手写数学符号转为标准LaTeX格式，用 $...$ 包裹。`

#### 1.3 AI 判断流程
- 文件：`src/app/api/guide/route.ts`
- 在 guide API 中增加参数 `studentImageUrl`（base64）
- 如果收到图片，先调用 student-image API 做识别，将识别结果作为 `studentInput` 传给 AI
- AI 正常判断对错并引导下一步（逻辑不变）

#### 1.4 前端交互流程
```
学生点击上传/粘贴图片 → 显示预览 → 点击发送
  → 前端发送 { studentImageUrl: base64, questionId, ... } 到 /api/guide
  → 后端识别图片 → 得到文本 → AI判断 → 返回回复
  → 对话框显示：学生消息显示缩略图 + AI回复
```

#### 1.5 注意事项
- 百度 OCR 对数学公式识别较差，尤其是分数、根号等，所以一定要有 qwen fallback
- 学生消息气泡要支持显示图片（目前只显示文字）
- 压缩图片到合理大小，避免 base64 过大导致请求超时

## 会话 2：左侧板书区（已引导通过的解题步骤）

### 目标
在解题页面的左侧区域，将 AI 已确认正确的步骤以板书形式展示，形成完整的解题过程记录。

### 技术方案

#### 2.1 数据结构设计
- 文件：`src/lib/types.ts`
- 新增类型：
```typescript
interface BoardStep {
  stepNumber: number;        // 步骤编号
  content: string;           // 步骤内容（LaTeX）
  isCorrect: boolean;        // 是否确认正确
  timestamp: number;         // 记录时间
}
```

#### 2.2 板书状态管理
- 文件：`src/app/solve/page.tsx`
- 新增 state：`const [boardSteps, setBoardSteps] = useState<BoardStep[]>([]);`
- **核心逻辑**：在 sendMessage 的 AI 回复处理中，检测到学生步骤被确认正确时，将该步骤添加到板书
- 判断依据：
  - 如果 AI 回复中包含正向信号（"很好"、"正确"、"没错"、"对的"、"就是这样"、"可以"）且不是最终完成
  - 同时提取 AI 回复中总结的该步骤结论（AI 在 prompt 中已经会总结）
  - 添加到 boardSteps

#### 2.3 AI Prompt 调整
- 文件：`src/app/api/guide/route.ts`
- 在 prompt 中增加规则：当学生回答正确时，在该步回复末尾用 `[STEP:步骤编号|步骤结论LaTeX]` 标记
  - 例如学生第一步算对了，AI回复末尾加 `[STEP:1|$f(x) = x^2 - 2x + 1$]`
- 前端解析 `[STEP:n|content]` 标记，自动提取到板书

#### 2.4 左侧布局调整
- 文件：`src/app/solve/page.tsx`
- 当前左侧只显示题目（约第683行），改为上下分区：
  - 上方：题目显示（不变）
  - 下方：板书区域（新增）
- 板书区域：
  - 标题："解题板书"
  - 每个步骤显示为卡片：`步骤1: xxx`
  - 步骤间用连接线或箭头连接
  - 空状态显示："完成正确步骤后将自动记录在这里"
  - 背景用浅色网格线，模拟黑板/稿纸效果
  - 步骤内容用 `<MathContent>` 渲染 LaTeX

#### 2.5 板书样式参考
```
┌─────────────────────────┐
│  题目                    │  ← 原有题目区域
│  ...                     │
├─────────────────────────┤
│  📝 解题板书             │  ← 新增板书区域
│                          │
│  ① $f(x) = x^2-2x+1$   │
│       ↓                  │
│  ② $f'(x) = 2x-2$      │
│       ↓                  │
│  ③ 令 $2x-2=0$ ...      │
│                          │
└─────────────────────────┘
```

#### 2.6 注意事项
- 板书只在 AI 确认正确后才添加，做错的不展示
- 切换到下一题时清空板书
- 板书区域需要有足够的滚动空间

## 会话 3：UI 美化 + 学习地图升级 + 分数系统

### 目标
提升整体视觉效果，让 Demo 更有冲击力。核心：让学生的学习成果可视化。

### 技术方案

#### 3.1 高考分值系统（核心新功能）
- 文件：`src/data/chapters.ts` 或新建 `src/data/chapter-scores.ts`
- 每个章节配置高考占分（参考高考数学大纲分值分布），例如：
  - 集合：5分，函数：12分，三角函数：15分，数列：8分，立体几何：12分……
  - 总分150分，所有章节分值之和应等于150
- 顶部统计区显示**"当前掌握 X / 150 分"**，用醒目的数字展示
- 章节卡片上显示该章节对应分值（如"三角函数 · 15分"）
- 学完章节后分值加到总分上

#### 3.2 分数衰减机制
- 文件：`src/lib/storage-v2.ts` 扩展
- 每个章节记录 `lastReviewTimestamp`（上次梳理时间）
- 衰减规则（示例，可调整）：
  - 7天内：分数颜色正常（绿色/金色），表示"记住了"
  - 7-14天：分数颜色变暗（黄色），提示"快忘了"
  - 14天以上：分数颜色消失（灰色），表示"遗忘了"
- 做一次知识梳理（/review），`lastReviewTimestamp` 更新，分数重新亮回来

#### 3.3 梳理深度等级
- 文件：`src/lib/storage-v2.ts` 扩展
- 每个章节记录 `reviewCount`（梳理次数）
- 梳理深度随次数递进：
  - 第0次（未梳理）：基础回顾，AI 带着过一遍知识点
  - 第1次：正常梳理，AI 引导回忆 + 查漏补缺
  - 第2次：深入梳理，AI 关联跨章节知识，比如做三角函数时关联函数的单调性
  - 第3次+：综合检验，AI 出综合题检验掌握程度
- 梳理页面的 prompt 根据 `reviewCount` 动态调整（修改 `/api/review/route.ts`）

#### 3.4 学习地图视觉升级
- 文件：`src/app/page.tsx`
- 章节卡片改造：
  - 已通关的章节显示知识点覆盖率（如"知识点覆盖 8/12（67%）"）
  - 分值用显眼的方式展示（如卡片右上角圆形标签）
  - 分数状态用颜色区分：正常=金色，衰减中=黄色，遗忘=灰色
  - 进行中的章节：脉冲动画边框（amber 色）
  - 未解锁：灰色 + 模糊
- 顶部统计区：
  - 主数字改为"当前掌握 X / 150 分"，大字号+粗体
  - 下方保留"完成题目"、"通关章节"、"连续学习天数"三个小卡片
- 章节卡片间连接线（CSS 伪元素）

#### 3.5 整体配色优化
- 主色保持 indigo 不变
- 按钮样式：hover 动画（scale 微放大 + shadow）
- 卡片：hover 微交互（translate-y + shadow 变化）

#### 3.6 解题页面美化
- 文件：`src/app/solve/page.tsx`
- 顶栏增加章节进度条（细线型，全宽）
- 对话气泡样式优化：AI/学生消息添加小图标
- 快捷按钮美化
- 通关页面：添加庆祝动画

#### 3.7 注意事项
- 分值数据可以根据实际高考大纲调整，不需要精确到1分，取整即可
- 衰减时间参数先定7天/14天，后续可以根据实际效果调
- 不要过度设计，保持简洁
- 确保移动端适配

## 产品定位（Demo帖核心素材）

### 目标用户
高考数学30-90分的学生。成绩越低，收获越大。他们不是不努力，是拿到题知道要用什么知识点，但不知道怎么一步步推导下去。

### 产品定位
**高考数学30-90分学生的AI陪练——帮你找到卡住的那一步**

"陪你一步步想"比"不给你答案"更有温度，也更准确。不给答案是手段，帮你推过去才是目的。

### 核心优势（四层）

1. **知道学生缺什么** — 推荐引擎追踪每个知识点覆盖情况，精准补漏
2. **搭台阶** — 大题拆小步，做对一步给下一步，不需要一步到位
3. **记住错误** — 不只记"错了"，还记"在第几步错了、错在哪"，复习时个性化错因追踪
4. **构建知识体系** — 章节通关+知识覆盖率+学习地图，帮学生建立有结构的数学知识网络

### 待开发功能（优先级排序）
- [x] 知识关联提示（融入会话2板书区，AI板书时顺带提示关联知识）
- [x] 高考分值系统 + 分数衰减（会话3，学习地图展示掌握分数）
- [x] 梳理深度等级（会话3，梳理次数递进深度）
- [ ] 入学诊断（5-8道精选题，生成初始画像，推荐学习路径）
- [ ] 高强度做题后的回顾过渡（不中断学习状态，切换到回顾模式"换档不停机"）

### 技术方案

#### 4.1 学习地图全部解锁

#### 4.2 首页：AI推荐章节（体现"知道学生缺什么"）
- 文件：`src/app/page.tsx`
- 在学习地图上方增加一个 **"今日推荐"** 区域，逻辑根据学生是否有进行中的章节分两种情况：

**情况一：有进行中的章节（存在 `unlocked && !isCleared && completedCount > 0` 的章节）**
- 不推荐新章节，显示**快速回顾昨天内容的入口**

- 如果有昨天的记录，显示："昨天学了[章节名]，快速回顾一下？" + "开始回顾"按钮
- 点击后跳转到该章节的做题页面，AI自动从昨天做到的地方继续
- 如果没有昨天的记录（第一次使用），显示："继续学习[章节名]" + "继续"按钮
- 这个推荐**不需要AI参与**，纯本地逻辑

**情况二：没有进行中的章节（所有已开始的章节都通关了，或第一次使用）**
- 显示3个章节推荐按钮：
  - **基础巩固**（简单）：推荐学生还没开始或覆盖率最低的简单章节（如集合）
  - **进阶提升**（中等）：推荐学生已开始但未通关的中等章节（如函数）
  - **挑战突破**（较难）：推荐学生已通关但梳理次数少的较难章节（如三角函数）
- 推荐逻辑（纯本地，不需要新API）：
  ```typescript
  // 简单：difficulty === "easy" 且 coveredKP 最少的未通关章节
  // 中等：difficulty === "medium" 且 coveredKP 最低的未通关章节
  // 较难：difficulty === "hard" 且 reviewCount 最少的已通关章节
  ```
- 每个推荐卡片显示章节名 + 分值 + 一句话理由（如"集合是数学基础，5分不能丢"）

- 文件：`src/app/solve/page.tsx`
- 当前流程：章节通关 → 显示"chapter-cleared"页面（第366-370行）
- 改为：章节通关 → 显示通关信息 + **推荐梳理按钮** → 学生可以选择"梳理知识点"或"返回地图"
- 梳理按钮跳转到 `/review?chapter=xxx`
- 如果该章节的 `reviewCount === 0`，显示提示："梳理一遍知识点可以巩固记忆，对应的分值也能保持住"
- 梳理页面已有（`/review`），梳理深度等级也已在会话3实现

#### 4.8 每日记忆检查（独立功能）
- 文件：新建 `src/app/api/daily-check/route.ts`
- 逻辑：学生进入网站时，检查已通关但超过7天未梳理的章节
- 从 `storage-v2.ts` 的 `getChapterProgress` 获取各章节的 `lastReviewTimestamp`
- 如果 `Date.now() - lastReviewTimestamp > 7 * 24 * 3600 * 1000`，则该章节有需要记忆检查的内容
- UI：在首页学习地图上方显示一个轻量提示条（可关闭）：
  - "三角函数已经7天没复习了，分值正在变暗。现在快速回顾一下？"
  - 两个按钮："快速回顾"（跳转到 review 页面）和"稍后再说"（关闭提示）
- 一次最多显示1个章节的提示，不要堆叠
- 每日只显示一次（用 sessionStorage 记录今天是否已显示）

#### 4.9 整体流程图（最终版）
```
进入网站
  → 首页：今日推荐（简/中/难）+ 记忆检查提示（如果有）
  → 学生选章节
  → 第一次进入：知识点预习 → 开始做题
  → 再次进入：直接做题
  → 做题中：
    → AI引导步骤（含知识关联提示）
    → 学生发文字/图片，AI判断对错
    → 板书记录正确步骤
    → 做完一题：回顾知识点 + 预告下一题方向
    → 每4题：小回顾（夸奖+覆盖率+提及错误）
  → 章节通关：推荐梳理
  → 返回地图：分数更新
```

#### 4.10 注意事项
- 被动式规划的核心原则：**学生感觉不到"被规划"**，一切都是自然发生的
- 小回顾是放松换档，不是考核，夸奖要多，语气要轻松
- 知识预习不要太长，3-5句话，不阻断学习热情
- 每日记忆检查不阻断学习流程，做成可关闭的提示条
- 知识关联提示不要每道题都提，只在确实有明显关联时才提，否则会干扰学生
- 所有新功能优先用本地逻辑实现，只有需要AI生成内容时才调API，控制成本
